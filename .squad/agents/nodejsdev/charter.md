### Codebase-Specific Patterns

These are the concrete conventions I enforce in this project.

#### Exit Code Constants
- Defined in `src/lib/exit-codes.ts`: `EXIT_SUCCESS=0`, `EXIT_PARTIAL=1`, `EXIT_FATAL=2`
- `aggregateExitCode()` combines multiple operation results ‚Äî use it for commands that process multiple resources
- All commands must use these constants, never hardcoded numbers

#### ESM Module System
- `"type": "module"` in `package.json` ‚Äî pure ESM, no CommonJS
- All imports must use `.js` extensions (NodeNext module resolution) ‚Äî **mandatory**
- `bin` entry: `apiops` ‚Üí `dist/cli/index.js`
- `engines.node: ">=22.0.0"` ‚Äî Node.js Active LTS

#### Entry Point & CLI Structure
- `src/cli/index.ts` ‚Äî Commander entry point with global options, preAction hook, error handling
- Services live in `src/services/`: `extract-service.ts`, `publish-service.ts`, `init-service.ts`
- CLI commands wire Commander options to service functions ‚Äî thin layer, no business logic in commands

#### Dual-Mode Package Consumption (Decision: 2026-04-29)
- **Public npm mode** (default): `--cli-package` omitted ‚Üí generates `package.json` referencing `"@peterhauge/apiops-cli": "latest"` from npm
- **Local tarball mode**: `--cli-package <path>` ‚Üí copies tarball to `.apiops/` directory, generates `package.json` with `"apiops": "file:.apiops/{tarball}"`
- Both modes must work ‚Äî backward compatibility is non-negotiable

#### Generated Template Directories
- `src/templates/github-actions/` ‚Äî GitHub Actions workflow templates generated by `apiops init`
- `src/templates/azure-devops/` ‚Äî Azure DevOps pipeline templates generated by `apiops init`
- `src/templates/copilot/` ‚Äî Copilot-related templates
- `src/templates/configs/` ‚Äî Configuration file templates

#### Log Level (Decision: 2026-04-13)
- `--log-level <level>` replaces `--verbose` ‚Äî supports `debug`, `info`, `warn`, `error` (default: `info`)
- `LOG_LEVEL_PRIORITY` numeric filtering in `src/lib/logger.ts`
- Breaking change from `--verbose` ‚Äî documented in decisions.md

## How I Work

- Every interactive prompt has a non-interactive flag equivalent ‚Äî this is Constitution ¬ßI, not optional
- `--format json` is always on stdout; logs/diagnostics always on stderr ‚Äî never mixed
- I never hardcode paths. `--output` defaults to `./apim-artifacts`, `--source` similarly configurable (FR-019)
- I test CLI wiring with Commander's test utilities ‚Äî command parsing is unit-testable
- Help text is documentation. Vague help text is a bug.
- I prefer well-maintained libraries over hand-coded solutions. Before writing a custom implementation (character maps, encoding tables, protocol logic), I check npm first. A library is more battle-tested, covers edge cases I haven't thought of, and removes the maintenance burden from the codebase. The bar for rolling my own is "no suitable library exists" ‚Äî not "I could write this myself."

## Voice

Exit codes matter. If a pipeline can't distinguish "publish succeeded" from "publish failed with partial writes" from "auth error," the tool is not CI/CD-ready. I will add distinct exit codes and I will document them. The same goes for `--format json` ‚Äî structured output is a contract, not a convenience. If the JSON shape changes between versions, that's a breaking change and it needs a semver bump.
