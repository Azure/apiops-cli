== Foundational Principles

=== gh-aw First

GitHub Agentic Workflows (gh-aw) are the **primary execution layer** for all repository operations that do not require human judgment.

* **Triage**: Every new issue is classified by the triage agent, labeled, and routed to the correct owner within minutes.
* **Monitoring**: The heartbeat workflow detects stalled work and re-engages agents automatically.
* **Notifications**: Status changes flow through GitHub's native notification system — no external tools required.
* **Escalation**: When an agent cannot resolve an issue, it escalates to a human maintainer with full context.

[NOTE]
====
Humans remain in the loop for: security-sensitive merges, architecture decisions, and release approvals. Everything else is agent-driven.
====

=== Security by Default

Defense in depth is applied at every layer:

[cols="1,2"]
|===
|Layer |Controls

|**Authentication**
|OIDC federation with Azure (no stored credentials); `GITHUB_TOKEN` scoped to minimum permissions per workflow

|**Dependencies**
|All Actions pinned to SHA; Dependabot enabled with auto-merge for patches; CodeQL scans daily

|**Secrets**
|No long-lived secrets in repo; environment-scoped secrets only; secrets never logged

|**Supply Chain**
|npm provenance on publish; `package-lock.json` committed; `npm ci` enforced in CI

|**Code**
|TypeScript strict mode; eslint with security rules; CodeReviewer agent checks §VIII (secret safety)
|===

=== Automation Over Manual Process

The automation principle is enforced through a simple heuristic:

> **If a human has to do it more than once, it becomes a workflow.**

Examples already applied:

* Issue classification → `squad-triage.yml`
* Label synchronization → `sync-squad-labels.yml`
* Progress monitoring → `squad-heartbeat.yml`
* Lint + test on every PR → `ci.yml`

Examples planned:

* Stale issue cleanup → `stale.yml`
* Dependency updates → Dependabot + auto-merge workflow
* Release publishing → tag-triggered npm publish with provenance

=== Contributor-Friendly

Contributors (human and AI) deserve:

* **Clear templates** — Issue and PR templates with structured fields
* **Fast feedback** — CI completes in under 3 minutes; triage bot responds in under 1 minute
* **Visible prioritization** — Labels communicate priority (`P0-critical` through `P3-wontfix`)
* **Documented flows** — This plan and `CONTRIBUTING.md` explain every path from idea to release

=== Spec-Driven Development

All features trace to `specs/001-apiops-cli/spec.md`. The traceability chain is:

----
Spec Requirement → Task (tasks.md) → GitHub Issue → Branch → PR → Merge → Release
----

* No code lands without a traceable requirement
* Tasks in `specs/*/tasks.md` are checked off as work completes
* The Squad AI team uses this chain to maintain context across agent handoffs

=== 3.2 Issue Triage (gh-aw IssueOps)

When a new issue is opened, a gh-aw workflow analyzes the content, classifies it, applies appropriate labels, and posts a triage comment explaining the classification.

.`.github/agents/issue-triage.yml`
[source,yaml]
----
---
on:
  issues:
    types: [opened]
permissions:
  contents: read
  issues: write
skip-bots: true
skip-author-associations: []
safe-outputs:
  add-labels:
    allowed: [bug, needs-info, enhancement, question, documentation, apim-api-change, type/spike, external-contribution]
    max: 2
  add-comment:
    max: 1
---

You are a triage agent for the `apiops-cli` repository — a TypeScript CLI tool
(`@peterhauge/apiops-cli`) for Azure API Management configuration-as-code.

Analyze the issue title and body. Classify the issue into ONE primary category:

- **bug**: The issue describes broken behavior, errors, or regressions in existing functionality (extract, publish, init commands, authentication, YAML parsing).
- **enhancement**: The issue requests new functionality or improvements to existing behavior.
- **question**: The issue is asking how to use apiops-cli or seeking clarification, not reporting a defect.
- **documentation**: The issue is about missing, incorrect, or unclear documentation.
- **needs-info**: The issue lacks sufficient detail to classify or reproduce. Missing version, no repro steps, or vague description.
- **apim-api-change**: The issue reports a change in the Azure APIM REST API that affects the tool.

Apply the most specific label. If the issue clearly falls into one category, apply only that label.
If the issue is unclear or missing critical information, apply `needs-info` as the second label.

Then post a triage comment in this format:

```
### 🏷️ Triage Classification

**Category:** <category>
**Reason:** <1-2 sentence explanation of why this classification was chosen>

<If needs-info was applied, list what specific information is missing>

---
_This issue was automatically triaged. A maintainer will review shortly._
```
----


=== 3.3 Issue Assignment (gh-aw)

When specific labels are applied to an issue, a gh-aw workflow automatically assigns it to the appropriate person or agent.

.`.github/agents/issue-assign.yml`
[source,yaml]
----
---
on:
  issues:
    types: [labeled]
permissions:
  contents: read
  issues: write
skip-bots: true
safe-outputs:
  assign-to-user:
    max: 1
  assign-to-agent:
    max: 1
  add-comment:
    max: 1
---

You are an assignment agent for `apiops-cli`. When an issue receives a label,
determine who should work on it based on these rules:

**Assignment Rules:**

1. `priority/critical` → assign-to-user: @peterhauge (project owner handles critical issues)
2. `type/spike` → assign-to-agent (spikes are well-suited for agent research)
3. `area/extract` or `area/publish` → assign-to-agent (core CLI logic)
4. `area/ci` → assign-to-user: @peterhauge (CI/infrastructure changes need human oversight)
5. `area/docs` or `documentation` → assign-to-agent (documentation tasks)
6. `external-contribution` → assign-to-user: @peterhauge (external PRs need human review)
7. `bug` with no area label → assign-to-agent (investigation and fix)
8. `enhancement` → do nothing (wait for prioritization)

**After assignment, post a brief comment:**

```
### 👤 Assignment

**Assigned to:** <user or agent>
**Reason:** <why this assignment was made based on the labels>
```

If no rule matches the current label event, take no action.

=== 3.4 Stale Issue Management (gh-aw Scheduled)

A weekly scheduled workflow identifies issues with no recent activity, warns contributors, and eventually closes abandoned issues.

.`.github/agents/stale-issues.yml`
[source,yaml]
----
---
schedule: weekly on monday
on:
  schedule: weekly on monday
permissions:
  contents: read
  issues: write
safe-outputs:
  add-labels:
    allowed: [stale]
    max: 20
  add-comment:
    max: 20
  close-issue:
    max: 10
  remove-labels:
    max: 20
---

You are a stale issue manager for `apiops-cli`. Your job is to keep the issue
tracker healthy by identifying and managing inactive issues.

**Rules:**

1. **Mark as stale (30 days inactive):**
   - Find open issues with no comments or activity in the last 30 days
   - Exclude issues with labels: `priority/critical`, `priority/high`, `pinned`, `stale` (already marked)
   - Add the `stale` label
   - Post this comment:
     ```
     ### ⏰ Stale Issue Notice

     This issue has had no activity for 30 days. It will be automatically closed
     in 30 days if there is no further activity.

     If this issue is still relevant:
     - Add a comment with an update or additional context
     - Remove the `stale` label

     _This is an automated message from the apiops-cli maintenance workflow._
     ```

2. **Close stale issues (60+ days inactive):**
   - Find issues labeled `stale` with no activity for 30+ additional days (60 total)
   - Exclude issues with labels: `priority/critical`, `priority/high`, `pinned`
   - Close the issue
   - Post this comment:
     ```
     ### 🔒 Closed — Stale

     This issue was automatically closed after 60 days of inactivity.
     If this is still relevant, please reopen with updated context.

     _Automated by apiops-cli maintenance._
     ```

3. **Do NOT mark or close:**
   - Issues with `priority/critical` or `priority/high` labels
   - Issues with the `pinned` label
   - Pull requests (only issues)

=== 3.5 Community Voting & Prioritization

GitHub Reactions (👍) on issues serve as the primary community voting mechanism. A weekly workflow scans for highly-voted issues and surfaces them for prioritization.

==== How Voting Works

* Users add 👍 reactions to issues they want prioritized
* Issues with >10 thumbsup receive the `high-demand` label automatically
* A weekly report summarizes the top-voted open issues

.`.github/agents/community-votes.yml`
[source,yaml]
----
---
schedule: weekly on monday
on:
  schedule: weekly on monday
permissions:
  contents: read
  issues: write
tools:
  github:
    toolsets: [default]
safe-outputs:
  add-labels:
    allowed: [high-demand]
    max: 10
  add-comment:
    max: 10
  create-project-status-update:
    max: 1
---

You are a community prioritization agent for `apiops-cli`. Every week you
scan open issues for community demand signals using GitHub reactions.

**Tasks:**

1. **Identify high-demand issues:**
   - Search all open issues for those with 10 or more 👍 (thumbsup) reactions
   - If an issue has ≥10 thumbsup and does NOT already have the `high-demand` label, add it
   - Post a comment on newly labeled issues:
     ```
     ### 🔥 High Community Demand

     This issue has received significant community interest (10+ 👍 votes).
     It has been flagged for prioritization review.

     _Automated by apiops-cli community voting workflow._
     ```

2. **Generate weekly prioritization report:**
   - List the top 10 open issues by thumbsup reaction count
   - Create a project status update with this format:
     ```
     ## 📊 Weekly Community Priorities — apiops-cli

     Top-voted open issues this week:

     | Rank | Issue | 👍 Votes | Labels |
     |------|-------|----------|--------|
     | 1 | #<num> — <title> | <count> | <labels> |
     | ... | ... | ... | ... |

     **New high-demand issues this week:** <count>
     **Total open high-demand issues:** <count>
     ```

3. **Remove stale high-demand labels:**
   - If an issue labeled `high-demand` now has fewer than 5 thumbsup (reactions removed),
     remove the `high-demand` label

=== 4.2 LabelOps Patterns

==== Pattern 1: `label_command:` (One-Shot Trigger)

The label is applied, the workflow fires, and the label is automatically removed. Used for imperative actions.

**Use Case:** The `deploy` label triggers an npm publish workflow.

.`.github/agents/labelops-deploy.yml`
[source,yaml]
----
---
on:
  issues:
    types: [labeled]
  pull_request:
    types: [labeled]
permissions:
  contents: read
  actions: write
label_command: deploy
safe-outputs:
  dispatch-workflow:
    max: 1
  add-comment:
    max: 1
  remove-labels:
    allowed: [deploy]
    max: 1
---

You are a deployment trigger for `apiops-cli`. When the `deploy` label is applied
to a merged PR on the main branch:

1. Verify the PR is merged into `main` (if not, post a comment explaining why deployment was skipped)
2. Dispatch the npm publish workflow via `dispatch-workflow`
3. Post a confirmation comment:
   ```
   ### 🚀 Deployment Triggered

   npm publish workflow dispatched for `@peterhauge/apiops-cli`.
   Monitor progress in the Actions tab.

   _Triggered by `deploy` label._
   ```

The `deploy` label will be automatically removed after this workflow completes (label_command behavior).
----

== Pull Request Lifecycle

=== 5.1 Internal PRs

==== Branch Naming Convention

All internal branches follow the pattern:

----
squad/{issue-number}-{slug}
----

Examples:

* `squad/42-init-command`
* `squad/87-fix-yaml-parsing`
* `squad/103-refactor-auth-module`

==== Required Checks Before Merge

Every internal PR must pass the following status checks before merge is permitted:

1. **Lint** — `npm run lint` (ESLint with TypeScript parser)
2. **Build** — `npm run build` (TypeScript 5.x strict compilation)
3. **Test** — `npm test` (vitest, all suites green)
4. **Coverage** — Line coverage ≥ 80%, no decrease from base branch

These are enforced as required status checks in the GitHub branch protection rule for `main`.

==== Squash Merge Policy

All internal PRs use **squash merge** to keep `main` history linear and bisectable.

* The squash commit message must summarize the entire branch's work (not just the last commit).
* Include `Closes #N` in the commit body to auto-close the linked issue.
* Co-authored-by trailers are preserved automatically.

==== Auto-Delete Branches

Branch auto-deletion is enabled in repository settings:

* After merge, the source branch (`squad/*`) is automatically deleted.
* This prevents stale branch accumulation.
* Contributors must not reuse deleted branch names for unrelated work.

=== 5.2 External PRs (Hostile Review)

==== Fork PR Isolation

External PRs from forks run in a restricted security context:

* **No secrets** — `secrets.*` context is empty for fork PRs.
* **Limited permissions** — `GITHUB_TOKEN` has read-only access.
* **No write access** — Cannot push to the base repository.
* **Artifact isolation** — Artifacts from fork PRs are sandboxed.

All external contributions require **at least one human maintainer approval** before merge. Automated approvals (bot accounts, gh-aw workflows) do not satisfy this requirement.

=== 5.3 Code Review Requirements

==== CodeReviewer Agent

The CodeReviewer agent reviews all PRs that modify files under `src/` or `tests/` against the project constitution (`.squad/identity/constitution.md`, §I–§VIII).

Review checks include:

* TypeScript strict compliance (§I)
* Testability and test coverage (§VI)
* Forward compatibility / extensibility (§VII)
* Secret safety — no credentials in source (§VIII)
* YAGNI — no speculative abstractions (§V)

==== Severity Levels

[cols="1,2,3",options="header"]
|===
| Level | Meaning | Action Required

| 🔴 Blocker
| Critical defect, security vulnerability, or constitution violation
| Must be resolved before merge. PR cannot proceed.

| 🟡 Required
| Significant issue that must be addressed
| Must be resolved before merge. Author revises.

| 🟢 Suggestion
| Improvement opportunity, style preference
| Author's discretion. May be deferred to follow-up issue.
|===

==== Blocker Resolution

* Any finding at 🔴 Blocker severity **blocks the PR from merge**.
* The author must push a fix commit and re-request review.
* The CodeReviewer re-evaluates only the blocker findings on re-review.

==== Reviewer Lockout Protocol

If a PR is **rejected** (reviewer requests changes with 🔴 Blocker findings):

* The original author may revise and push fixes.
* If the same author's revision is rejected a second time, the PR is reassigned to a different team member.
* The rejected author cannot self-approve their own revision.
* A fresh reviewer must approve the final state before merge.

=== 5.4 Test Requirements

==== When Tests Are Required

[cols="2,2,2",options="header"]
|===
| Change Type | Required Tests | Notes

| Any change to `src/`
| Unit tests required
| Tests must cover new/modified code paths

| New CLI command
| Unit tests + integration test
| Integration test exercises full command lifecycle

| Bug fix
| Regression test required
| Test must fail without the fix, pass with it

| Refactor (no behavior change)
| Existing tests must pass
| No new tests required unless coverage drops below threshold

| Documentation only
| None
| No test requirement

| Dependency updates only
| Existing tests must pass
| No new tests required
|===

==== Coverage Threshold

* **Minimum:** 80% line coverage across the project
* **No decrease allowed:** PR coverage must be ≥ base branch coverage
* **New files:** Must individually meet 80% line coverage
* **Enforcement:** Coverage check is a required status check

==== Coverage Tooling

* **Runner:** vitest with built-in coverage via `@vitest/coverage-v8`
* **Reporter:** `text`, `lcov`, `json-summary` (for CI parsing)
* **Configuration:**

==== Coverage Enforcement

* **Minimum threshold:** 80% lines (enforced by vitest `thresholds` — build fails if not met)
* **No decrease rule:** Compared against base branch coverage via PR comment
* **PR comment:** Coverage summary posted automatically using gh-aw `add-comment`

==== Coverage PR Comment Workflow

[source,yaml]
----
# .github/workflows/coverage-comment.yml
name: Coverage Report

on:
  workflow_run:
    workflows: ["CI"]
    types: [completed]

permissions:
  pull-requests: write
  actions: read

jobs:
  comment:
    if: >
      github.event.workflow_run.event == 'pull_request' &&
      github.event.workflow_run.conclusion == 'success'
    runs-on: ubuntu-latest
    steps:
      - name: Download coverage artifact
        uses: actions/download-artifact@fa0a91b85d4f404e444e00e005971372dc801d16 # v4.1.8
        with:
          name: coverage-report
          run-id: ${{ github.event.workflow_run.id }}
          github-token: ${{ secrets.GITHUB_TOKEN }}

      - name: Parse coverage summary
        id: coverage
        run: |
          LINES=$(jq '.total.lines.pct' coverage/coverage-summary.json)
          BRANCHES=$(jq '.total.branches.pct' coverage/coverage-summary.json)
          FUNCTIONS=$(jq '.total.functions.pct' coverage/coverage-summary.json)
          echo "lines=$LINES" >> "$GITHUB_OUTPUT"
          echo "branches=$BRANCHES" >> "$GITHUB_OUTPUT"
          echo "functions=$FUNCTIONS" >> "$GITHUB_OUTPUT"

      - name: Post coverage comment
        uses: actions/github-script@60a0d83039c74a4aee543508d2ffcb1c3799cdea # v7.0.1
        with:
          script: |
            const lines = '${{ steps.coverage.outputs.lines }}';
            const branches = '${{ steps.coverage.outputs.branches }}';
            const functions = '${{ steps.coverage.outputs.functions }}';
            const status = parseFloat(lines) >= 80 ? '✅' : '❌';

            const body = `## ${status} Coverage Report

            | Metric | Coverage |
            |--------|----------|
            | Lines | ${lines}% |
            | Branches | ${branches}% |
            | Functions | ${functions}% |

            **Threshold:** 80% lines minimum. No decrease allowed.`;

            // Find the PR number from the workflow run
            const prs = await github.rest.pulls.list({
              owner: context.repo.owner,
              repo: context.repo.repo,
              state: 'open',
              head: context.payload.workflow_run.head_branch
            });

            if (prs.data.length > 0) {
              await github.rest.issues.createComment({
                owner: context.repo.owner,
                repo: context.repo.repo,
                issue_number: prs.data[0].number,
                body: body
              });
            }

=== Auto-Merge Policy

[cols="1,2,2",options="header"]
|===
| Update Type | Action | Rationale

| Patch (x.x.PATCH)
| Auto-approve + auto-merge if CI passes
| Low risk, bug fixes and security patches

| Minor (x.MINOR.x)
| Add `needs-review` label, require human review
| May introduce new features or subtle behavior changes

| Major (MAJOR.x.x)
| Add `breaking-change` label + create evaluation issue
| High risk, may require code changes

=== Dependabot Auto-Merge Workflow

[source,yaml]
----
# .github/workflows/dependabot-auto-merge.yml
name: Dependabot Auto-Merge

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  contents: write
  pull-requests: write
  issues: write

jobs:
  evaluate:
    if: github.actor == 'dependabot[bot]'
    runs-on: ubuntu-latest
    steps:
      - name: Fetch Dependabot metadata
        id: metadata
        uses: dependabot/fetch-metadata@d7267f607e9d3fb96fc2fbe83e0af444e2ba7c86 # v2.3.0
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}

      # --- Patch updates: auto-approve and auto-merge ---
      - name: Auto-approve patch updates
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr review --approve "$PR_URL"
        env:
          PR_URL: ${{ github.event.pull_request.html_url }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Enable auto-merge for patch updates
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr merge --auto --squash "$PR_URL"
        env:
          PR_URL: ${{ github.event.pull_request.html_url }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      # --- Minor updates: label for review ---
      - name: Label minor updates for review
        if: steps.metadata.outputs.update-type == 'version-update:semver-minor'
        uses: actions/github-script@60a0d83039c74a4aee543508d2ffcb1c3799cdea # v7.0.1
        with:
          script: |
            await github.rest.issues.addLabels({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              labels: ['needs-review']
            });

      # --- Major updates: label + create evaluation issue ---
      - name: Handle major updates
        if: steps.metadata.outputs.update-type == 'version-update:semver-major'
        uses: actions/github-script@60a0d83039c74a4aee543508d2ffcb1c3799cdea # v7.0.1
        with:
          script: |
            const pr = context.payload.pull_request;
            const depName = '${{ steps.metadata.outputs.dependency-names }}';
            const newVersion = '${{ steps.metadata.outputs.new-version }}';
            const prevVersion = '${{ steps.metadata.outputs.previous-version }}';

            // Add breaking-change label to PR
            await github.rest.issues.addLabels({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: pr.number,
              labels: ['breaking-change']
            });

            // Create evaluation issue
            const issueBody = `## Major Dependency Update Evaluation

            **Package:** \`${depName}\`
            **Update:** ${prevVersion} → ${newVersion}
            **PR:** #${pr.number}

            ### Checklist

            - [ ] Review changelog/migration guide for \`${depName}\`
            - [ ] Identify breaking changes that affect our codebase
            - [ ] Update code to accommodate breaking changes
            - [ ] Verify all tests pass with the new version
            - [ ] Update any affected documentation

            ### Decision

            - [ ] ✅ Approve — merge the Dependabot PR
            - [ ] ❌ Reject — close the PR, pin current version
            - [ ] ⏸️ Defer — revisit in next sprint

            ---
            _Auto-generated by Dependabot major update handler_`;

            await github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: `[Evaluate] Major update: ${depName} ${prevVersion} → ${newVersion}`,
              body: issueBody,
              labels: ['breaking-change', 'dependencies']
            });

== Supply Chain Security

=== npm Audit in CI

Every CI run includes a security audit step:

[source,yaml]
----
      - name: Security audit
        run: npm audit --audit-level=high
----

* Audit runs after `npm ci` to check the resolved dependency tree.
* CI fails on **high** or **critical** severity vulnerabilities.
* Moderate/low findings are tracked but do not block merge.
* `npm audit fix` PRs are created manually after evaluation.

=== package-lock.json Integrity

[cols="1,3",options="header"]
|===
| Rule | Rationale

| Always use `npm ci`
| Ensures reproducible installs from lockfile; fails if lockfile is out of sync with `package.json`

| Never use `npm install` in CI
| `npm install` can mutate the lockfile, introducing non-deterministic builds

| Lockfile committed to source
| `package-lock.json` is always committed; `.gitignore` must not exclude it

| Lockfile changes require review
| Any PR that modifies `package-lock.json` must be inspected for unexpected dependency additions
|===

=== GitHub Action Pinning

All GitHub Actions are pinned to **full commit SHA**, not tags or branch names:

[source,yaml]
----
# ✅ CORRECT — pinned to full SHA
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2

# ❌ WRONG — tag can be force-pushed
- uses: actions/checkout@v4

# ❌ WRONG — branch is mutable
- uses: actions/checkout@main
----

* The comment after the SHA records the semantic version for human readability.
* Dependabot automatically proposes SHA updates when new action versions are released.
* This prevents supply chain attacks via compromised action tags.

=== No postinstall Scripts from Dependencies

Dependencies that execute `postinstall` (or `preinstall`, `install`) scripts are a supply chain risk vector.

==== Audit Hook

Add a CI step that detects lifecycle scripts from dependencies:

[source,yaml]
----
      - name: Audit lifecycle scripts
        run: |
          # List all packages with install scripts
          SCRIPTS=$(npm query ':attr(scripts, [postinstall])' 2>/dev/null || echo "[]")
          if [ "$SCRIPTS" != "[]" ] && [ -n "$SCRIPTS" ]; then
            echo "::warning::Dependencies with postinstall scripts detected:"
            echo "$SCRIPTS" | jq -r '.[].name' 2>/dev/null || true
          fi
----

==== .npmrc Configuration

[source,ini]
----
# .npmrc
ignore-scripts=true
----

* `ignore-scripts=true` prevents lifecycle scripts from running during `npm ci`.
* If a specific package legitimately requires a postinstall script (e.g., native bindings), it must be explicitly allowlisted and documented in this maintenance plan.

== Security

=== 9.1 Threat Model

==== Threat Actors

[cols="1,3",options="header"]
|===
| Actor | Capabilities & Motivations

| **Maintainer**
| Trusted with merge rights, access to repository secrets. Can publish releases. Threat: Compromised account, insider threat.

| **Squad Agent**
| Automated agent with write access to branches, read access to issues/PRs. Cannot merge without approval. Threat: Prompt injection, scope creep.

| **External Contributor**
| Fork workflow, no write access, no secrets. Submits PRs from public forks. Threat: Malicious code submission, pwn request attacks.

| **Dependency Author**
| Upstream npm package maintainer. Publishes updates pulled by Dependabot. Threat: Supply chain attack via compromised dependency.

| **GitHub Actions Runner**
| Ephemeral VM executing CI/CD workflows. Isolated per run. Threat: Workflow injection, secret exfiltration.
|===

==== Attack Surface

[cols="2,1,3",options="header"]
|===
| Vector | Risk Level | Mitigation

| **Pwn Request (malicious fork PR)**
| HIGH
| Fork PRs run without secrets, `pull_request` not `pull_request_target`, human review required

| **Action tag hijack**
| MEDIUM
| All Actions pinned to full SHA, not tags or branches

| **Workflow injection**
| HIGH
| No user-controlled input in `run:` blocks, `${{ }}` expressions sanitized

| **Dependency confusion**
| MEDIUM
| Scoped package `@peterhauge/apiops-cli`, lockfile committed, `npm ci` enforced

| **Lockfile tampering**
| MEDIUM
| CODEOWNERS review required for `package-lock.json` changes, integrity verified by `npm ci`

| **Secret exfiltration**
| CRITICAL
| Secrets scoped to environments, never logged, no `echo` of secret vars, fork PRs have no secret access

| **Governance tampering (`.github/` modification)**
| CRITICAL
| CODEOWNERS protection, human approval required, gh-aw monitoring of protected paths
|===


=== 9.2 Mitigations

==== Workflow Hardening

* **`pull_request` trigger for fork code** — External PRs use `pull_request` (not `pull_request_target`) to ensure fork code runs in an isolated context without access to repository secrets.
* **Secret scoping** — Secrets are scoped to GitHub Environments (`integration-test` environment) and only workflows explicitly granted access can read them.
* **Permission minimization** — Every workflow declares minimal `permissions:` at job level. Default is `permissions: read-all` downgraded to `contents: read` unless write access is explicitly required.
* **No dynamic `run:` blocks with user input** — Workflow steps never interpolate user-controlled data (issue titles, PR bodies, comments) directly into `run:` commands. Use `actions/github-script` with sanitized inputs instead.

Example:
[source,yaml]
----
# ✅ SAFE — user input is passed as an argument, not interpolated
- uses: actions/github-script@60a0d83039c74a4aee543508d2ffcb1c3799cdea
  with:
    script: |
      const title = context.payload.issue.title;
      console.log("Issue title:", title);

# ❌ UNSAFE — direct interpolation enables injection
- run: echo "${{ github.event.issue.title }}"
----

==== Dependency Hardening

* **SHA-pinned Actions** — All third-party GitHub Actions are pinned to full commit SHA with a comment noting the semantic version:
  `uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2`
* **`npm ci` enforcement** — CI and publish workflows use `npm ci`, never `npm install`. `npm ci` fails if `package-lock.json` is out of sync with `package.json`, ensuring reproducible builds.
* **Lockfile integrity** — `package-lock.json` is committed and protected by CODEOWNERS. Any PR modifying it must be reviewed for unexpected additions.
* **`ignore-scripts=true`** — `.npmrc` disables lifecycle scripts (`postinstall`, `preinstall`) from dependencies to prevent supply chain attacks via script execution.

==== Review Hardening

* **CODEOWNERS for `.github/`** — Workflow files, Actions configurations, and CI/CD infrastructure require approval from `@peterhauge` (project owner).
* **Mandatory human review for external PRs** — All PRs from forked repositories must be approved by at least one human maintainer. Bot approvals do not satisfy this requirement.
* **CodeReviewer agent** — Automated review for constitution compliance (§I–§VIII), including secret safety checks (§VIII). Blockers must be resolved before merge.

==== AI/Agent Safety

* **No automated merges** — Squad agents and bots can approve PRs, but they cannot merge. All merges require explicit human action.
* **Branch-only writes** — Agents create branches (`squad/*`) and open PRs but cannot push directly to `main`.
* **Human approves all releases** — Publishing to npm (tag creation, `deploy` label) requires human initiation. Agents cannot trigger releases.
* **Prompt injection mitigation** — Agent prompts are parameterized; issue titles and bodies are treated as data, not executable instructions.

            \`\`\`

            \`\`\`

=== 9.5 Incident Response

==== Compromised Dependency Playbook

If a dependency is found to be compromised (backdoor, malware, or unexpected behavior):

1. **Immediate Actions (within 1 hour):**
   - Create a `priority/critical` issue with title: `[Security] Compromised dependency: <package-name>`
   - Pin the affected dependency to the last known-good version in `package.json`
   - Run `npm install` to update the lockfile
   - Open an emergency PR with title: `fix: pin <package> to last known-good version (security)`
   - Merge after expedited review (CodeReviewer + one human maintainer)

2. **Short-Term (within 24 hours):**
   - Audit all published versions of `@peterhauge/apiops-cli` that include the compromised dependency
   - If any published version is affected, publish a **patch release** with the pin
   - Notify users via GitHub Security Advisory and npm deprecation message

3. **Long-Term (within 1 week):**
   - Evaluate alternatives to the compromised package
   - If no alternative, consider vendoring the last known-good version
   - Document the incident in a post-mortem issue

==== Leaked Secret Playbook

If a secret (API key, token, password) is accidentally committed or exposed:

1. **Immediate Actions (within 15 minutes):**
   - **Revoke the secret immediately** at the source (npm, Azure, GitHub)
   - Rotate the secret and update it in GitHub Secrets
   - Create a `priority/critical` issue: `[Security] Secret exposure: <type>`

2. **Short-Term (within 1 hour):**
   - Use `git filter-repo` or BFG Repo-Cleaner to remove the secret from Git history
   - Force-push the cleaned history to all branches (if applicable)
   - Notify all collaborators to `git pull --rebase` and delete local copies

3. **Long-Term (within 24 hours):**
   - Audit access logs for the revoked secret (was it used maliciously?)
   - Implement pre-commit hooks or `git-secrets` to prevent future leaks
   - Document the incident and update `SECURITY.md` with prevention guidance

==== Malicious PR Merged Playbook

If a malicious PR is accidentally merged to `main`:

1. **Immediate Actions (within 30 minutes):**
   - Revert the merge commit: `git revert -m 1 <merge-commit-sha>`
   - Open an emergency PR with the revert, merge immediately
   - Lock the `main` branch temporarily (disable merges)
   - Create a `priority/critical` issue: `[Security] Malicious PR merged: #<num>`

2. **Short-Term (within 2 hours):**
   - Audit what the malicious code did (file access, network calls, environment inspection)
   - Check if any secrets were accessed during CI runs
   - Rotate all repository secrets and npm tokens as a precaution
   - If the malicious code was published to npm, **unpublish the affected version immediately**

3. **Long-Term (within 1 week):**
   - Strengthen review process: require multiple approvals for external PRs
   - Audit CODEOWNERS and branch protection rules
   - Document the incident in a post-mortem and share lessons learned with the team

==== Deployment Failure Incident

When the npm publish workflow fails (detected by `deployment_status` event):

1. **Automated Response:**
   - The `deploy-failure-handler.yml` workflow automatically creates an incident issue labeled `incident`, `deployment-failure`
   - Issue includes: deployment environment, ref, failure description, and log URL

2. **Manual Investigation (within 1 hour):**
   - Review deployment logs to identify root cause
   - Common causes: version mismatch, CHANGELOG missing, CI test failure, npm token expired

3. **Resolution:**
   - Fix the underlying issue (update package.json, fix tests, rotate npm token)
   - Re-tag the release or apply the `deploy` label to retry
   - Once a successful deployment completes to the same environment, the incident issue auto-closes

=== End-to-End Flows (Prose)

==== Bug Report Flow

1. **Report**: Contributor opens an issue using the bug report template, providing reproduction steps, expected behavior, and environment details (Node.js version, OS, apiops-cli version).
2. **Triage**: Within minutes, `squad-triage.yml` fires. The gh-aw triage bot classifies the issue as `bug`, assigns a priority label (`P0` through `P2`), and an area label (`area/cli`, `area/extract`, `area/publish`).
3. **Assignment**: `squad-issue-assign.yml` routes the issue to the appropriate Squad agent based on area expertise.
4. **Implementation**: The assigned agent creates a feature branch (`fix/<issue-number>-<slug>`), implements the fix with tests, and opens a PR referencing `Closes #N`.
5. **CI Validation**: `ci.yml` runs eslint, vitest, and tsc. All checks must pass.
6. **Code Review**: The CodeReviewer agent reviews against the constitution (§I–§VIII). Blockers must be resolved.
7. **Human Approval**: A maintainer performs final review and merges to `main`.
8. **Release**: Maintainer creates a semver tag. The publish workflow builds with provenance and pushes to npm as `@peterhauge/apiops-cli`.
9. **Verification**: Post-publish, the heartbeat workflow confirms the new version is accessible on npm.

==== Feature Request Flow

1. **Idea**: Contributor opens an issue using the feature request template describing the use case.
2. **Triage**: gh-aw bot labels as `enhancement` and assigns priority based on alignment with `specs/001-apiops-cli/spec.md`.
3. **Spec Update**: If approved, the Tech Lead adds the requirement to the spec and creates tasks in `specs/001-apiops-cli/tasks.md`.
4. **Issue Decomposition**: Tasks are broken into GitHub issues with `[T0XX]` identifiers.
5. **Implementation**: Squad agents implement task-by-task, each PR referencing its issue.
6. **Integration**: Once all tasks for a feature are complete, a release is cut.

==== External Contribution Flow

1. **Fork & Branch**: External contributor forks the repo and creates a branch.
2. **Development**: Contributor follows `CONTRIBUTING.md` guidelines — TypeScript strict mode, vitest tests, eslint compliance.
3. **PR Submission**: Contributor opens a PR. The gh-aw bot greets them with guidelines and expected timelines.
4. **CI (Sandboxed)**: CI runs without secrets — only lint, type-check, and unit tests. Integration tests are skipped for external PRs.
5. **Review Cycle**: Human maintainer reviews, provides feedback. Contributor iterates.
6. **Merge**: After approval from both the human maintainer and CodeReviewer agent, the PR is merged.
7. **Attribution**: Contributor is credited in the commit history. No CLA is required (MIT license).

==== Emergency Hotfix Flow

1. **Detection**: A critical bug is reported (security vulnerability, data loss, or complete CLI failure).
2. **Escalation**: Issue is immediately labeled `P0-critical`. Heartbeat workflow alerts all maintainers.
3. **Hotfix Branch**: A `hotfix/<issue>` branch is created directly from the latest release tag.
4. **Minimal Fix**: Only the critical fix is applied — no bundled features or refactors.
5. **Expedited Review**: CodeReviewer agent + one human maintainer. Normal review SLA is waived.
6. **Immediate Release**: Tag is created and npm publish is triggered within hours of the fix landing.
7. **Backport**: Fix is cherry-picked to `main` if the hotfix branch diverged from HEAD.
8. **Post-Mortem**: A brief post-mortem is added to the issue documenting root cause and prevention measures.
