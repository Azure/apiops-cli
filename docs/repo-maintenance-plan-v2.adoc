== 1. Executive Summary

This repository maintenance plan governs the **Azure API Operations CLI** project, providing automation patterns for issue triage, label management, contributor workflows, CI/CD pipelines, and code quality gates. All automation is implemented using **gh-aw** (GitHub Agentic Workflows), which orchestrates YAML-defined workflows triggered by GitHub events.

**Core Automation Domains:**

* **Issue Lifecycle Management**: Automated triage, assignment, stale detection, and community voting
* **Label Operations**: Pattern-based label management using LabelOps
* **CI/CD Integration**: Build, test, lint, and publish pipelines with quality gates
* **Code Quality**: Coverage verification, static analysis, and dependency auditing
* **Documentation**: Freshness checks and automated updates
* **APIM Integration**: Azure API Management change detection and deployment

== 2. Foundational Principles

=== 2.1 Automation-First Philosophy

**Principle**: Every repeatable task SHOULD be automated. Manual intervention is a last resort for exceptional cases.

**Application**:

* Issue triage runs automatically on every new issue
* Label changes trigger downstream workflows (e.g., `status:needs-triage` → auto-assign triager)
* Scheduled workflows handle recurring maintenance (stale issues, community reports, dependency updates)

=== 2.2 Transparency & Auditability

**Principle**: All automated actions MUST be visible in issue/PR timelines with clear attribution.

**Application**:

* Every workflow uses `github-actions[bot]` identity or scoped bot accounts
* Comments explain WHY an action occurred (e.g., "Marked stale after 30 days of inactivity")
* Workflow runs appear in Actions tab with full logs

=== 2.3 Graceful Degradation

**Principle**: Automation failures MUST NOT block human workflows.

**Application**:

* Workflows use `continue-on-error: true` for non-critical steps
* Failed workflows post visible comments requesting manual intervention
* Labels like `automation:failed` trigger escalation to maintainers

=== 2.4 Community-Centric Design

**Principle**: Automation SHOULD empower contributors, not gatekeep.

**Application**:

* Issue templates guide contributors to provide necessary context
* Automated triage provides immediate feedback (expected timeline, next steps)
* Community voting (`👍` reactions) influences roadmap prioritization

=== 3.2 Automated Issue Triage

**Workflow**: `.github/workflows/issue-triage.yml`

**What It Does**:

. Detects new issues via `issues.opened` trigger
. Parses issue body using `template_data` safe-output
. Applies labels based on template fields (e.g., `priority:high` if severity is "Critical")
. Assigns default triager from `CODEOWNERS`
. Posts welcome comment with expected response time

**gh-aw Pattern**: IssueOps (https://github.github.com/gh-aw/patterns/issue-ops/)

**Key Safe-Outputs**: `template_data`, `assign-to-user`, `add-labels`

**Configuration**: Triage rules are defined in YAML using `when:` conditionals. No embedded scripts required.

=== 3.3 Issue Assignment

**Workflow**: `.github/workflows/issue-assign.yml`

**What It Does**:

. Listens for `/assign @username` or `/assign @squad-agent` comments
. Validates commenter has `write` permissions or is issue author
. Assigns issue to specified user OR invokes Squad agent
. Updates `assignee` field and posts confirmation comment

**gh-aw Pattern**: IssueOps + LabelOps (https://github.github.com/gh-aw/patterns/label-ops/)

**Key Safe-Outputs**: `assign-to-user`, `assign-to-agent`, `comment`

**Example**: `/assign @copilot` routes issue to Copilot CLI agent for autonomous resolution.

=== 3.4 Stale Issue Management

**Workflow**: `.github/workflows/stale-issues.yml`

**What It Does**:

. Runs weekly via `schedule: cron('0 0 * * 0')`
. Identifies issues with no activity for 30 days → applies `status:stale` label + warning comment
. Identifies issues with no activity for 60 days → closes with explanation
. Excludes issues with `pinned`, `priority:high`, or `status:blocked` labels

**gh-aw Pattern**: ProjectOps (https://github.github.com/gh-aw/patterns/project-ops/)

**Key Safe-Outputs**: `add-labels`, `close-issue`, `comment`

**Policy Override**: Contributors can remove `status:stale` label to prevent auto-close.

=== 3.5 Community Voting

**Workflow**: `.github/workflows/community-votes.yml`

**What It Does**:

. Runs weekly to count `👍` reactions on `type:feature` issues
. Issues with ≥10 reactions receive `priority:community-driven` label
. Generates weekly report posted to Discussions (roadmap planning)
. Integrates with project board to prioritize high-demand features

**gh-aw Pattern**: IssueOps + ProjectOps (https://github.github.com/gh-aw/patterns/issue-ops/, https://github.github.com/gh-aw/patterns/project-ops/)

**Key Safe-Outputs**: `query-issues`, `add-labels`, `create-discussion`

**Rationale**: Democratizes roadmap by surfacing community priorities.

=== 4.2 LabelOps Patterns

**What is LabelOps?** A gh-aw pattern that triggers workflows when labels are added/removed. See https://github.github.com/gh-aw/patterns/label-ops/ for details.

**Two Primary Patterns**:

1. **One-Shot Commands** (`label_command:`): Executes action once when label is applied, then removes label
2. **Persistent State** (`names:`): Continuously enforces rules while label is present

**Pattern 1: One-Shot Assignment**

When `status:needs-assignment` label is added, assign to default squad agent and remove label.

**What It Does**:

* Trigger: `label_command: status:needs-assignment`
* Action: Call `assign-to-agent` safe-output with `agent: @copilot-cli`
* Cleanup: Remove `status:needs-assignment` label
* Result: Issue assigned to agent, label disappears

**Pattern 2: Persistent Enforcement**

While `priority:critical` label is present, enforce 4-hour SLA monitoring.

**What It Does**:

* Trigger: `names: [priority:critical]`
* Action: Run hourly check, post escalation comment if no activity for 4 hours
* Persistence: Runs on every check UNTIL label is removed
* Result: Continuous SLA monitoring

**Reference**: See https://github.github.com/gh-aw/patterns/label-ops/ for full syntax and advanced examples.

== 5. Pull Request Lifecycle

=== 5.1 Internal PRs

*Branch Naming:* `squad/{issue-number}-{slug}` (e.g., `squad/42-init-command`)

*Required Checks Before Merge:*

* Lint — `npm run lint` (ESLint with TypeScript parser)
* Build — `npm run build` (TypeScript 5.x strict compilation)
* Test — `npm test` (vitest, all suites green)
* Coverage — Line coverage ≥ 80%, no decrease from base

*Merge Strategy:* Squash merge to `main`. Squash commit message must summarize entire branch work, include `Closes #N` to auto-close issue.

*Branch Cleanup:* Auto-delete source branch after merge.

=== 5.2 External PRs (Hostile Review)

*Fork Isolation:*

* No secrets exposed (`secrets.*` context empty)
* `GITHUB_TOKEN` has read-only access
* No write access to base repository
* Artifacts sandboxed

*Human Review Required:* At least one human maintainer approval before merge. Bot approvals don't satisfy this requirement.

*Contribution Guidelines Comment:* A workflow (`pull_request_target`) auto-comments on fork PRs with CLA requirements, test expectations, and local validation checklist. See https://github.github.com/gh-aw/events/pull_request_target/[gh-aw pull_request_target docs].

*CLA:* Microsoft CLA bot (`microsoft/cla`) checks CLA status. PRs from unsigned contributors are blocked.

=== 5.3 Code Review Requirements

*CodeReviewer Agent:* Reviews all PRs modifying `src/` or `tests/` against project constitution (`.squad/identity/constitution.md`, §I–§VIII).

*Review Checks:*

* TypeScript strict compliance (§I)
* Testability and coverage (§VI)
* Forward compatibility / extensibility (§VII)
* Secret safety (§VIII)
* YAGNI — no speculative abstractions (§V)

*Severity Levels:*

[cols="1,2,3",options="header"]
|===
| Level | Meaning | Action Required

| 🔴 Blocker
| Critical defect, security vulnerability, constitution violation
| Must resolve before merge. PR blocked.

| 🟡 Required
| Significant issue
| Must resolve before merge. Author revises.

| 🟢 Suggestion
| Improvement opportunity
| Author's discretion. May defer to follow-up.
|===

*Lockout Protocol:* If a PR is rejected (🔴 findings) twice by the same reviewer, reassign to a different team member. Original author cannot self-approve.

== 6. CI/CD & Quality

=== 6.1 CI Pipeline

*Triggers:* Pull requests targeting `main`, pushes to `main` (post-merge verification).

*Matrix:* 3 OS (ubuntu-latest, windows-latest, macos-latest) × Node 22.x. Fail-fast disabled.

*Steps:*

1. Checkout (`actions/checkout@SHA`)
2. Setup Node.js 22.x (`actions/setup-node@SHA` with `cache: npm`)
3. Install dependencies (`npm ci`)
4. Lint (`npm run lint`)
5. Build (`npm run build`)
6. Test with coverage (`npx vitest run --coverage`)
7. Upload coverage artifact (ubuntu-latest only, 7-day retention)

*Caching:* `node_modules` cached via `package-lock.json` hash. Handled by `setup-node` action. Cache miss triggers full `npm ci`.

=== 6.2 Code Coverage

*Tooling:* vitest + `@vitest/coverage-v8`. Reporters: `text`, `lcov`, `json-summary`.

*Thresholds:* 80% lines, 75% branches, 80% functions/statements. Build fails if not met (vitest `thresholds` config).

*PR Comment:* A `workflow_run` workflow downloads coverage artifact, parses `coverage-summary.json`, and posts a table to the PR. Uses gh-aw safe output pattern with `actions/github-script@SHA`. See https://github.github.com/gh-aw/reference/safe-outputs/[safe outputs reference].

*No Decrease Rule:* PR coverage must be ≥ base branch coverage (enforced by CodeReviewer agent and manual review).

=== 6.4 npm Publish / Deployment

*Triggers:*

1. Git tag push matching `v*` (e.g., `v0.2.0`)
2. LabelOps `deploy` label on release issue (one-shot, label removed after trigger). See https://github.github.com/gh-aw/patterns/label-ops/[LabelOps pattern].

*Pre-Publish Checks (enforced by workflow):*

1. Version in `package.json` matches git tag
2. `CHANGELOG.md` contains entry for release version
3. All tests pass with coverage ≥ 80%
4. Lint passes with zero errors
5. TypeScript build succeeds

*Publish Command:* `npm publish --access public --provenance`

* `--provenance` enables SLSA Level 2 attestation (see §7.2)
* Uses `NPM_TOKEN` secret with publish permissions
* Package: `@peterhauge/apiops-cli` on https://registry.npmjs.org

*Deployment Failure Handler:* A workflow on `deployment_status` creates an incident issue (labels: `incident`, `deployment-failure`) with investigation checklist. Auto-closes older incidents for same environment.

=== 7.2 Supply Chain Security

*npm Audit in CI:*

Every CI run includes:

[source,yaml]
----
- name: Security audit
  run: npm audit --audit-level=high
----

* Runs after `npm ci` to check resolved dependency tree
* Fails on high/critical vulnerabilities (blocks merge)
* Moderate/low findings tracked but don't block
* `npm audit fix` PRs created manually after evaluation

*Lockfile Integrity:*

*Rules:*

* Always use `npm ci` in CI — ensures reproducible installs, fails if lockfile out of sync
* Never use `npm install` in CI — can mutate lockfile, causes non-deterministic builds
* `package-lock.json` committed to source (not gitignored)
* Lockfile changes require review — inspect for unexpected dependency additions
* Lockfile is single source of truth for dependency resolution

*GitHub Action Pinning:*

All actions pinned to **full commit SHA**, not tags or branches:

[source,yaml]
----
# ✅ CORRECT — pinned to full SHA
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2

# ❌ WRONG — tag can be force-pushed
- uses: actions/checkout@v4
----

* Comment after SHA records semantic version for human readability
* Dependabot proposes SHA updates when new action versions released
* Prevents supply chain attacks via compromised action tags

*No postinstall Scripts:*

*.npmrc Configuration:*

[source,ini]
----
# .npmrc
ignore-scripts=true
----

* Prevents lifecycle scripts (`postinstall`, `preinstall`, `install`) from running during `npm ci`
* Dependencies with install scripts are supply chain risk vectors
* If specific package requires postinstall (e.g., native bindings), must be explicitly allowlisted and documented

*CI Audit Hook:* Optional step detects packages with lifecycle scripts via `npm query ':attr(scripts, [postinstall])'` and logs warnings.

*SLSA Compliance:*

*npm Provenance:* `npm publish --provenance` provides SLSA Level 2 build provenance:

* Source tied to specific git commit via GitHub Actions OIDC
* npm generates signed attestation linking package to CI workflow run
* Consumers verify with `npm audit signatures`

=== 7.3 APIM REST API Change Detection

A scheduled workflow monitors the Azure API Management REST API for changes that may require updates to `apiops-cli`. This automated detection helps stay aligned with APIM platform evolution and ensures compatibility with new API versions.

*Schedule and Trigger:*

The workflow executes every 6 hours via `gh-aw` link:https://github.github.com/gh-aw/reference/triggers/[`schedule:` trigger]. This frequency balances responsiveness with resource efficiency, catching significant API changes within a reasonable window.

*Monitoring Sources:*

The detector scans three key sources for APIM changes:

* *Azure REST API Specs Repository:* `Azure/azure-rest-api-specs/specification/apimanagement`
  - Recent commits and merged PRs (last 6 hours)
  - Focus on resource-manager API specification changes
* *Azure APIM Changelog:* link:https://learn.microsoft.com/en-us/azure/api-management/api-management-whats-new[What's New in API Management]
* *Stable API Version Releases:* New versions like `2024-05-01`, `2024-06-01`, etc.

*Detection Criteria:*

Issues are automatically created when any of these conditions are detected:

* New stable API version released
* Breaking changes to core endpoints: `/apis`, `/products`, `/policies`, `/backends`, `/namedValues`, `/diagnostics`, `/authorizationServers`, `/openidConnectProviders`
* Deprecated endpoints or properties
* New resource types that `apiops-cli` should support
* Schema changes to existing resource types

*Issue Creation and Output:*

When changes are detected, the workflow uses the `create-issue` link:https://github.github.com/gh-aw/reference/safe-outputs/[safe output] with `deduplicate-by-title: true` to prevent duplicate issues for the same change. The auto-created issue includes:

* *Title Format:* `[APIM Change] <api-version>: <brief-description>`
* *Labels:* `apim-api-change`, `status/triage`, and relevant area (`area/extract` or `area/publish`)
* *Content:*
  - API version and change type (new-version, breaking-change, deprecation, new-resource, schema-change)
  - Detailed description of what changed
  - Affected `apiops-cli` areas (extract, publish, models, auth)
  - Source link (commit, PR, changelog entry)
  - Recommended action for implementing support

If no changes are detected, the workflow takes no action.

== 8. Documentation Freshness

*Purpose:* Detect when documentation drifts from source code to ensure accuracy and maintain trust with users.

*Schedule:* Weekly on Friday via `gh-aw` link:https://github.github.com/gh-aw/reference/triggers/[`schedule:` trigger] (cron: `0 12 * * 5`).

*Doc-to-Source Mapping:*

The workflow maintains a mapping table of documentation files to their corresponding source code:

[cols="2,3",options="header"]
|===
| Documentation File | Source Dependencies

| `README.md`
| `src/cli/index.ts`, `src/commands/*/index.ts`

| `docs/extract.md`
| `src/commands/extract/**`

| `docs/publish.md`
| `src/commands/publish/**`

| `docs/init.md`
| `src/commands/init/**`

| `docs/auth.md`
| `src/auth/**`, `src/azure/**`
|===

*Detection Criteria:*

Documentation is flagged as potentially stale when:

1. Source files modified more recently than corresponding docs (by >7 days)
2. Changes are significant:
   - New exported functions, classes, or commands
   - Changed function signatures or command options
   - Removed public APIs
   - Modified error messages or output formats

*Analysis Method:*

The workflow uses AST analysis (via TypeScript compiler API) to detect structural changes rather than cosmetic edits. Comments-only changes don't trigger alerts.

*Output:*

When stale documentation is detected, the workflow creates an issue using the `create-issue` link:https://github.github.com/gh-aw/reference/safe-outputs/[safe output] with `deduplicate-by-title: true` to prevent duplicate alerts:

* *Title:* `[Docs] Update needed: <doc-file>`
* *Labels:* `type/docs`, `status/needs-triage`
* *Content:*
  - Which documentation file needs updating
  - Which source files changed
  - Summary of detected changes
  - Link to source file diff

If all documentation is current, no action is taken.

|Coverage threshold |Untested code |CI: Vitest ≥80%
|CodeQL scan |Security issues |CodeQL workflow: blocks on high-severity findings
|Lockfile sync |Version mismatches |CI: `npm ci` fails if lockfile out of sync
|npm publish |Accidental/malicious publish |Human-only: maintainer with 2FA + token
|===

=== 9.5 Incident Response

*Compromised Dependency:* Run `npm audit` locally. Assess exploitability in CLI context. Patch via `npm update <package>` or remove dependency. If critical and in production, merge patch immediately and publish emergency release. Notify users via GitHub Security Advisory.

*Secret Leaked in Commit:* Revoke leaked secret immediately. Assess exposure (history, logs, artifacts). Scrub history via `git rebase -i` (unreleased) or `git filter-repo` (on `main`). Audit access logs. Update constitution §VIII and add pre-commit secret detection.

*Malicious PR Merged:* `git revert <sha>` immediately. Yank npm version if published. Check for runner/Azure compromise. Rotate GitHub Actions secrets and npm token. Identify bypass vector (compromised account, social engineering, injection). Implement additional gates (2+ approvals, MFA, workflow approval). Publish Security Advisory.

*Security Contact:* Follow https://aka.ms/SECURITY.md. Report vulnerabilities privately, not via GitHub issues.

=== 11.3 Glossary

**gh-aw**:: GitHub Agentic Workflows - a framework for declarative GitHub automation using YAML-defined workflows

**IssueOps**:: A pattern where issue comments trigger automated workflows (e.g., `/assign`, `/label`)

**LabelOps**:: A pattern where label changes trigger automated workflows (e.g., adding `priority:critical` starts SLA monitoring)

**ProjectOps**:: A pattern for automating project board management (e.g., auto-move cards based on PR status)

**Safe Outputs**:: Pre-defined gh-aw outputs that safely interact with GitHub API without custom scripts (e.g., `assign-to-user`, `add-labels`)

**Frontmatter**:: YAML metadata block at the top of workflow files defining triggers, permissions, and configuration

**CODEOWNERS**:: GitHub file defining default reviewers for file paths

**Stale Issue**:: Issue with no activity for defined period (30 days warning, 60 days auto-close)

**Community-Driven Priority**:: Feature request with ≥10 👍 reactions, indicating high community demand
