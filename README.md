# apiops-cli

A CLI tool for Azure API Management (APIM) configuration-as-code workflows. Extract, publish, and manage APIM configurations from the command line.

## Prerequisites

- **Node.js** ≥ 22 (Active LTS)
- **npm** (ships with Node.js)

## Dev Container (Recommended)

The fastest way to get a fully configured development environment is to use the included [Dev Container](https://containers.dev/).

### Tools included

| Tool | Notes |
|------|-------|
| Node.js 22.x | Project runtime target |
| npm | Ships with Node.js |
| Git | Source control |
| GitHub CLI (`gh`) | Issue and PR management |
| Azure CLI (`az`) | Azure resource management |

### VS Code (local or Codespaces)

1. Install the [Dev Containers extension](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers).
2. Open the repository in VS Code.
3. When prompted, click **Reopen in Container** (or run **Dev Containers: Reopen in Container** from the Command Palette).
4. VS Code will build the image, install all tools, and run `npm ci` automatically.
5. Verify the environment:

   ```bash
   node --version   # should print v22.x.x
   az --version     # Azure CLI version
   gh --version     # GitHub CLI version
   ```

### GitHub Codespaces

Click **Code → Codespaces → Create codespace on \<branch\>** in the GitHub UI. The container starts pre-configured — no local setup needed.

## Getting Started

```bash
# Install dependencies
npm install

# Run the CLI in development mode (no build step required)
npm start
```

## Development Workflow

All commands assume you are in the repository root.

### Build

Compiles TypeScript to JavaScript in the `dist/` directory:

```bash
npm run build
```

### Lint

Runs ESLint with TypeScript-aware rules:

```bash
npm run lint
```

### Test

Runs the full test suite with Vitest:

```bash
npm test
```

Run tests in watch mode during development:

```bash
npm run test:watch
```

### Recommended Workflow Before Pushing

```bash
npm run lint && npm run build && npm test
```

Run all three in order. If any step fails, fix the issue before pushing.

## Producing an Executable

Since this is a CLI tool, there are two ways to run it locally:

### 1. Development Mode (no build needed)

Uses `tsx` to run TypeScript source directly:

```bash
npm start
```

### 2. Compiled Build

Build first, then run the compiled entry point:

```bash
npm run build
node dist/cli/index.js
```

### 3. Link as a Global Command

To install `apiops` as a command available anywhere on your system:

```bash
npm run build
npm link
apiops
```

To unlink later:

```bash
npm unlink -g apiops
```

## Testing Pipelines

Until the `apiops` package is published to npm, generated CI/CD pipelines need a local copy of the CLI to install. The `apiops init` command handles this via the `--cli-package` flag, which embeds an npm tarball directly in the target repository.

### Step 1 — Build and Pack the CLI

From the apiops-cli repository root:

```bash
npm run build
npm pack
```

This produces a tarball in the current directory, e.g. `apiops-0.1.0.tgz`.

### Step 2 — Initialize a New APIM Repository

Create (or `cd` into) the target repository where your APIM artifacts and pipelines will live, then run `apiops init`:

```bash
# GitHub Actions
apiops init \
  --cli-package /path/to/apiops-0.1.0.tgz \
  --ci github-actions \
  --environments dev,prod

# Azure DevOps
apiops init \
  --cli-package /path/to/apiops-0.1.0.tgz \
  --ci azure-devops \
  --environments dev,prod
```

> **Tip:** Add `--non-interactive` for scripted/CI usage. Without it the CLI will prompt for any missing options.

> **File conflicts:** If any of the target files already exist, the command will list the conflicts and exit with an error. Pass `--force` to overwrite existing files.

The init command will:

1. Copy the tarball into `.apiops/apiops-0.1.0.tgz` inside the target repo.
2. Generate a `package.json` with a local file dependency:
   ```json
   { "dependencies": { "apiops": "file:.apiops/apiops-0.1.0.tgz" } }
   ```
3. Generate CI/CD pipeline files (extract + publish workflows).
4. Generate configuration templates (`configuration.extract.yaml`, `configuration.<env>.yaml`).
5. Output an identity setup guide for configuring Azure authentication.

### Step 3 — Commit and Push

```bash
git add .
git commit -m "chore: initialize APIM repository with apiops pipelines"
git push
```

The generated pipelines use `npm ci` (which installs from the local tarball) and `npx apiops` — no npm registry access required. This works in air-gapped environments and across both GitHub Actions and Azure DevOps.

### Updating the CLI

To pick up new CLI changes, rebuild and re-pack in the apiops-cli repo, then re-run `apiops init --cli-package --force` in the target repo. The `--force` flag allows the existing files to be overwritten with the updated version.

### Migration to npm (future)

Once the `apiops` package is published to npm, the migration is a one-line change in the target repo's `package.json`:

```diff
- "apiops": "file:.apiops/apiops-0.1.0.tgz"
+ "apiops": "^0.1.0"
```

Then delete the `.apiops/` directory. Pipeline templates need no changes — `npm ci` + `npx apiops` continues to work.
